# Global rules — common (all apps)

Canonical source for the shared `app='global'` tier of the harness `rules` table. These
rules are **stack-neutral** and apply to every participating app regardless of framework.
Seeded only by `harness-infra/script/build_global_rules.rb`. Each `## Heading` is a
category; each bullet is one rule; severity is inferred from the wording
(never/must/always/do-not/non-negotiable → critical).

Stack-specific universal rules live in the sibling files `global-rails.md`,
`global-expo.md`, `global-godot.md`, and `global-unreal.md`, each seeded into its own
`global-<stack>` tier.

## Workflow

- Prefer removing obsolete paths outright over keeping deprecated compatibility layers. If code, docs, scripts, or workflows are no longer the intended path, delete or rewrite them directly unless there is a strong, explicit reason to preserve compatibility.
- Prefer what the framework or engine already provides over bypassing it. If a bypass is necessary, document the reason in code comments.
- Provide a direct status update in chat when implementation is complete or attention is required. Include completion/blocked state and the exact next action needed from the user when applicable.
- No repository-managed cross-repo handoff docs. Summarize cross-repo work directly in the task/PR/thread context.
- Use the Explore subagent to offload research tasks — it runs separately and returns only a summary, not raw file contents.
- Use `| tail -N` or `| grep` when running commands that produce verbose output.
- Work inside the running devcontainer whenever the repo ships one. If a repository provides a devcontainer (a `.devcontainer/` directory), always run the app's own toolchain — its tests, generators, builds, linters, dev server, and database commands — inside the running devcontainer rather than on the host, so the host environment stays clean and tooling versions stay identical to CI and to the other agents; start or attach to the devcontainer before you begin, and never run that toolchain directly on the host when a devcontainer exists. This governs the app's toolchain only — the host-side harness flow stays on the host: the git worktree/push/PR/teardown steps (which add and remove a sibling worktree outside the repo and run `git`/`gh` from the primary checkout, not from inside the container) and the few host-only harness scripts that document their own host requirement, such as the Linear archiver `harness-infra/script/archive_done_linear_tickets.rb`, all remain host-side. The infra already supports the container path: `harness_env.sh` detects host-vs-container via `/.dockerenv`, and once the app's devcontainer joins `agent-harness-net` its in-container tooling reaches the shared harness DB at `harness-db:5432` (that network join is the companion onboarding step a new devcontainer app must complete — see the integration runbook). This rule is keyed on the presence of a `.devcontainer/`, not on the app or the stack — an app that ships no devcontainer is unaffected, and any app that adds one, of any stack, inherits it automatically.
- Never commit directly to `main` or develop on it — all work happens on feature branches, and agent worktree branches merge into their feature branch. Merging a feature branch (or its reviewed PR) into `main` is a self-verified checklist, not a request for consent: when all three gate conditions hold — (a) an Opus agent has reviewed the ticket and signed off, (b) no open questions or issues remain for the user, and (c) CI is green on GitHub — an agent (including AI agents) must merge automatically as the default action; passing the gate IS the authorization, so do not pause to ask the user. Confirm each condition is actually true (verify the sign-off exists and CI is green now on GitHub, not merely assumed) — that is fact-checking, not consent-seeking. Pause only when a specific gate condition genuinely fails; never merge unreviewed, CI-failing, or open-question work into `main`. `main` is the production branch.
- Start every task by creating your own dedicated git worktree and branch — unconditionally, as the very first step, no permission needed and no condition to check; never work in the shared primary checkout, and always create your own worktree even if you believe you are the only agent. The harness shares one checkout, so a concurrent `git checkout` or branch switch by another agent silently moves your HEAD and working tree — commits then land on the wrong branch and uncommitted edits collide; a dedicated worktree is the only thing that prevents this, so it is the expected default, not something to defer or ask about. Run `git worktree add <path> -b <branch>`, do all work there, and tear it down at completion per the teardown rule below. Never assume the primary checkout's current branch is still yours across turns; re-verify the branch before staging or committing, and never run a destructive `git reset`/`checkout`/`clean` against a checkout another agent may be using.
- Push your own branch and open its pull request as soon as that branch has its first commit — automatically, as a mechanical step of the process with no decision in it and no permission needed, exactly like creating the worktree at the start; being assigned the task IS the authorization to push and open the PR, just as passing the merge gate IS the authorization to merge, so never leave a branch local-only and never wait for the user to say "push it" or "open a PR". Run `git push -u origin <branch>` and `gh pr create --base <the branch this one merges into>`, and link the task's tracker ticket in the PR body. Always pass `--base` explicitly: a bare `gh pr create` targets the repository's default branch, which would aim a worktree branch's PR at `main` and contradict the rule above that worktree branches merge into their feature branch. This binds whichever agent is responsible for a branch that carries commits — the implementer for its own worktree branch, and the agent holding a feature branch for that feature branch, including when the commits on it arrived by merging subtask branches rather than by its own hand; a feature branch is never exempt for want of an author. It is never a licence to push over another agent's in-flight branch, and it never loosens a strictly read-only review constraint that forbids `push`: an agent acting as a reviewer does not push, and hands the ticket back to an implementer instead. An unpushed branch is not cautious work held back, it is work nobody can act on: a reviewer cannot review what is not on GitHub and CI cannot produce the green signal the auto-merge gate depends on, so the gate can never pass and the task stalls behind an agent nobody knows is waiting. Push and PR come before that gate and are never gated by it — do not withhold them because review, CI, or open questions are still outstanding, since publishing the branch is precisely what lets those conditions be evaluated at all. Announcing that you could push, or offering to open a PR if the user says the word, is the same defect as not pushing — a hedged offer is a withheld PR; never report a task as complete-but-unpushed. If the change genuinely is not ready for review, open a draft with `gh pr create --draft` and mark it ready later — a draft PR is the answer to unfinished work, never a local-only branch. Two narrow carve-outs and nothing else: a repo with no remote configured is a real decision — ask the user which remote to use, never invent or add one unilaterally, and record the unpushed branch as a work-log todo until they answer; and a host guardrail that denies a command (the auto-mode permission classifier sometimes denies `gh pr merge`) constrains only the command it actually blocked and must never be read backwards as a reason to withhold the push or the PR — when merge is blocked the branch is still pushed and the PR is still open, and the blocked merge is the only thing you surface to the user.
- Tear down your worktree and branch when work completes — always, as the closing step of a task, the mirror of creating them at the start; a finished task leaves behind zero worktrees and zero branches of its own. Once your change has merged (into its feature branch, or into `main` via the auto-merge gate) or been abandoned, first confirm the work actually landed — by its squash commit in `main` or its merged PR (e.g. `gh pr view <branch> --json state,mergedAt`), never by `git branch --merged`, which reports a false negative here because the harness squash-merges every PR, so a merged branch's tip is never an ancestor of `main`. Then, running from the primary checkout (not from inside the worktree you are removing, which would block its own removal), remove your worktree with `git worktree remove <path>` (never leave uncommitted files behind), delete the branch with `git branch -D <branch>`, and run `git worktree prune`. The force flag `-D` is required, not a shortcut: plain `git branch -d` gates on that same ancestor check and will refuse to delete a squash-merged branch — so verify the merge out of band first, then force-delete. The only branches and worktrees that may persist are those holding genuinely in-flight, unmerged work; a merged or abandoned branch or a stale worktree left for another agent to puzzle over is a defect, not a convenience.
- Our focus is on writing high quality code the first time no matter how long that takes. We don't need to move fast.
- Keep docs updated when behavior changes. Update the project's documentation — README plus any architecture, design, runbook, API, or roadmap docs — whenever behavior, interfaces, runtime env vars, the data model, or critical workflows change. (Stack-specific apps may pin exact filenames in their stack or app tier.)
- New agent rules and guardrails must be saved to the harness, never to ad-hoc or file-based memory. Add app-specific rules to your app's `docs/harness/<app>-rules.md` (or role docs) and rebuild your app tier; propose universal or stack rules by editing the canonical files in `harness-infra/rules/`. The harness is the canonical rule store.
- Always read and update the harness work log. Read current context at the start of any work — latest resume note, open TODOs, recent done in `agent_harness.work_log` (via the SessionStart hook, `script/session_context.sh read`, or an MCP query). Before finishing or handing off, write back: `log`/`done` completed items, `add-todo` new work, `set-resume` a fresh handoff note. The harness work log — not chat scrollback or ad-hoc docs — is the canonical home for work-to-do / done / resume.
- Always complete all work, or explicitly track what you defer — never leave a surfaced issue in limbo. If something comes up during a task (a review finding, a bug, a missed edge case, dead code, a follow-up) and there is any question of whether to address it now (its severity, scope, or priority), either fix it in the current change or record it before finishing: a work-log todo (`add-todo`) for process/working items, and — for any code-facing issue — a ticket in the project's issue tracker (see the next rule). Raising it only in chat does not count as tracking it.
- Every surfaced issue gets a tracker ticket — always, no exceptions. Any bug, defect, review finding, regression, security gap, missed edge case, confirmed dead code, or follow-up identified during work must be filed as an issue in the project's canonical issue tracker (Linear, for the apps that use it) — even when it is pre-existing, low-priority, or out of scope for the current change. A work-log todo or a chat mention may supplement the ticket but never substitutes for it: deferred code work lives in a ticket. If that create fails because the shared Linear workspace is at its ticket cap, you must not drop the ticket or abandon the task — reclaim space with the shared archiver `harness-infra/script/archive_done_linear_tickets.rb` and retry the create (see the archiver-recovery rule below). Link the ticket from the relevant PR and work-log entry.
- A Linear ticket create that fails because the shared workspace is at its ticket cap (a limit-exceeded / "reached the issue limit" error; the cap is workspace-wide across every team, not per board) is never terminal and never grounds to give up — you must not skip the mandatory ticket or abandon the task. Treat it as required error recovery: reclaim space by archiving already-completed (Done) tickets with the shared archiver `harness-infra/script/archive_done_linear_tickets.rb`, then retry the create. Run it from the host — it needs only Ruby, internet, and `LINEAR_API_KEY` (which it reads from your login shell rc, e.g. `~/.zshrc`, when the key isn't already exported). Preview with `--dry-run` first, and scope with `--team KEY` / `--project NAME|ID` when targeting a specific board. Because the cap is workspace-wide, an unscoped run reclaims the most room; the per-team `--keep` / `--threshold` defaults only archive from teams that are individually crowded, so if a pass frees nothing, widen the scope (drop the `--team`/`--project` filter, lower `--threshold`, or force `--all` with `--yes`) rather than repeating the identical run. Do not retry blindly in a loop: if a real archive pass frees no space because no Done tickets remain to archive, stop retrying and escalate — record the ticket's full title and body in a work-log `add-todo` and flag it in chat for the user to resolve, so the required ticket is deferred, not lost. Run the same archiver proactively whenever the workspace nears its limit; and if you are in a devcontainer or CI and cannot run the host archiver, apply that same defer-and-flag fallback rather than silently dropping the ticket.
- The archiver only ever touches tickets already in a completed state, so it never removes active work; do not fork a per-app copy of it — the single shared script in `harness-infra` serves every team because the Linear ticket limit is shared across the whole workspace.
- Never report Linear as unavailable, unauthenticated, or "not connected" on the basis of a name lookup. Linear reaches this workspace through a **claude.ai connector whose MCP tools are prefixed with an opaque UUID, not the string `linear`** — currently `mcp__b1538edf-803f-40f0-8119-2805750db5a3__*`. A search of the tool registry for "linear", or an exact-name call to `mcp__linear__*`, will therefore find nothing even while Linear is fully working, and a session-start notice naming a `linear` server as requiring authentication refers to a retired duplicate, not to the connector you actually have. Two further traps: the issue-creating tool is **`save_issue`** (a create-or-update tool), not `create_issue` — no tool by that name exists — and comments are `save_comment`; and MCP tool schemas are deferred, so a call issued before `ToolSearch` loads the schema fails with a not-found error that looks identical to a missing server. Before concluding Linear is unreachable you must (a) resolve the real tool by capability rather than by name — `ToolSearch` for `save_issue` / `list_teams`, or select the UUID-prefixed tool directly — and (b) actually invoke it and quote the error it returns. "No tool matched my search" is not evidence the server is down; it is evidence you searched for the wrong name. Only a real invocation returning a real transport or authorization error justifies the claim, and even then the mandatory-ticket rules above still apply — defer and flag, never silently drop.
- Distinguish transport failure from application-level rejection before you report any tool failure, and never let a prompt or handoff instruct an agent to stop at the first error. A create that fails on the workspace ticket cap, a missing `teamId`, a validation error, or a rate limit is a *rejection by a working service* and is recoverable on the spot; only a genuine connection or authorization error is unavailability. Report which one you hit, and quote the error verbatim rather than paraphrasing it into a diagnosis. When you author a subagent prompt or handoff, do not bake in escape hatches of the form "if the tool fails, stop and report UNAVAILABLE" — that instruction manufactures the false negative it claims to guard against; tell the agent to diagnose and recover instead.
- Global rule tiers are centrally owned in `harness-infra/rules/` and seeded only by `harness-infra/script/build_global_rules.rb`. The `global` tier applies to every app; the `global-rails`, `global-expo`, `global-godot`, and `global-unreal` tiers apply to apps of that stack (selected at read time via `HARNESS_STACK`). Never seed or rewrite any `global*` tier from an app seeder — app seeders write only their own `app=<name>` rows and must abort if `HARNESS_APP` starts with `global`. To change a universal or stack rule, edit the canonical file in `harness-infra/rules/` and re-run the central seeder.

## Architecture

- Apply SOLID and clean-design principles — single responsibility, clear boundaries, dependency inversion. Prefer small, well-named units over large multi-purpose ones.
- Never swallow errors silently. Every caught exception must be reported or logged explicitly, with enough context to diagnose it — never discard an error and continue. (The specific reporting mechanism is a stack concern.)

## Testing

- All tests must pass before finishing work. Run the project's full test suite and any relevant linting before handoff or commit. Any failures — whether caused by your changes or pre-existing — must be fixed. Fix all warnings and errors in tests, build output, type checks, and runtime logs. Never silence or defer without a documented exception.
- Prefer correctness over speed. If a code change breaks existing tests, investigate why before adjusting anything. Do not weaken a test to preserve a green run.
- Never skip, hide, or remove failing tests to force a green run. A breaking test may be exposing a previously hidden bug. Fix the root cause in the code under test or the test's incorrect assumptions. Do not delete an agent-written test to make it pass — debug the failure and rewrite incorrect assertions; only remove a test if it is truly a duplicate or covers behavior that no longer exists.
